# Текстовая RPG в терминале

## Цель проекта

Сделать текстовую RPG-игру, которая работает прямо в терминале — без графики, без браузера, просто запустил и играешь. Хотелось попробовать сделать что-то интерактивное и при этом разобраться, как работают игровые движки изнутри.

---

## Задачи

- Придумать формат, в котором удобно описывать историю игры
- Сделать интерфейс прямо в терминале (без GUI)
- Реализовать систему статов персонажа (здоровье, сила, золото)
- Добавить инвентарь
- Сделать систему сохранений (несколько слотов)
- Сделать так, чтобы некоторые действия можно было сделать только один раз
- Реализовать условия на действия — например, "нужен меч чтобы атаковать"

---

## План

```
1. Выбрать формат хранения сценария → JSON
2. Написать загрузчик и парсер сценария
3. Реализовать движок (узлы, действия, условия, эффекты)
4. Сделать интерфейс на curses
5. Добавить сохранения
6. Протестировать на реальном сценарии
```

## Как устроен сценарий

Вся история хранится в JSON-файле. Каждый **узел** — это сцена с текстом и набором действий:

```json
"crossroads": {
  "text": "Ты стоишь на перепутье.",
  "actions": [
    {
      "text": "Идти налево",
      "next": "forest",
      "conditions": { "health": { ">=": 5 } },
      "effects": { "health": -2 }
    }
  ]
}
```

Это значит — **история не зашита в код**, можно написать любой сценарий не трогая Python вообще.

## Реализованные фичи

### Движок сцен
- Игра состоит из **узлов** (сцен) и **действий** между ними
- Каждое действие может вести в другой узел

### Система условий
```python
# Действие доступно только если выполнено условие
{ "strength": { ">=": 10 } }       # нужна сила 10+
{ "inventory_has": "ключ" }         # нужен предмет в инвентаре
```
Недоступные действия показываются с `[✗]` и выделяются красным

### Система эффектов
```python
{ "health": -5 }                    # получить урон
{ "money": 10 }                     # получить золото
{ "inventory_add": ["меч"] }        # добавить предмет
{ "inventory_remove": ["ключ"] }    # убрать предмет
```

### Одноразовые действия (`once: true`)
Некоторые действия можно сделать только один раз. После — показывается другой текст (`repeat_text`) и применяются другие эффекты (`repeat_effects`). Например, "Обыскать сундук" → второй раз "Сундук уже пуст"

### Система смерти
Если `health <= 0` — автоматически переход на узел `death` или на кастомный `death_next` из действия

### Сохранения
- 3 слота сохранений
- Сохраняются: текущий узел, все статы, инвентарь, выполненные действия
- При старте — меню выбора слота

### Интерфейс (curses)
- Панель статов с полоской здоровья `████░░░░░░░░`
- Прокрутка инвентаря стрелками `←/→`
- Выделение выбранного пункта меню инверсией цвета
- Обработка маленького окна терминала

## Почему именно такие решения

**JSON для сценария** — можно написать историю без знания Python. Геймдизайнер и программист могут работать отдельно.

**curses вместо просто `print()`** — обычный print не позволяет перерисовывать экран, двигать курсор и читать стрелки. curses даёт полный контроль над терминалом.

**ООП (классы)** — разделение на `Node`, `Action`, `SaveGame`, `GameEngine` позволяет не запутаться когда кода становится много. Каждый класс отвечает за своё.

**Узловая структура** — классический подход для текстовых квестов (так устроены почти все игры жанра). Граф узлов легко расширять.

Получился полноценный движок для текстовых RPG. Главное — **сценарий полностью отделён от кода**. Можно написать любую историю в JSON и она заработает без изменений в Python. Движок обрабатывает условия, эффекты, инвентарь, смерть и сохранения автоматически.
